# OneMarkDotNet 调试指南

## 1. 概述

COM Add-In 的调试与传统 .NET 应用程序有显著不同。OneMarkDotNet 作为 OneNote 的 COM Add-In，其调试面临以下特殊性和常见困难：

- **进程托管**：Add-In 代码运行在 OneNote 进程（ONENOTE.EXE）中，无法直接以独立进程启动调试
- **注册依赖**：Add-In 必须通过 RegAsm 注册到 Windows 注册表后，OneNote 才能加载它
- **COM 注册完整性**：RegAsm 只注册基本 COM 类型信息，OneNote 加载 .NET COM Add-In 还需要额外的 AppID 注册，否则会报"运行错误"
- **COM 生命周期**：OneNote 控制 Add-In 的加载和卸载时机，调试器必须在 OneNote 启动后附加
- **动态类型陷阱**：COM 互操作大量使用 `dynamic`，编译时无法检测类型错误，运行时才暴露问题
- **异常传播**：COM 边界会改变异常的行为，未捕获的异常可能导致 OneNote 直接禁用 Add-In
- **热重载限制**：修改代码后必须注销 → 重新构建 → 重新注册 → 重启 OneNote，无法热重载

本指南将详细说明如何克服这些困难，高效地调试 OneMarkDotNet。

## 2. 注册 COM Add-In

### 2.1 完整注册流程（三步）

OneNote COM Add-In 的注册需要三个步骤，缺一不可：

```
步骤 1: RegAsm /codebase    →  注册 COM 类型（CLSID、ProgId、InprocServer32 等）
步骤 2: 导入 register_full.reg  →  补充 AppID 等关键注册项（RegAsm 不会生成）
步骤 3: 重启 OneNote         →  OneNote 读取注册表加载 Add-In
```

> **重要**：如果只执行步骤 1 而跳过步骤 2，OneNote 会报"未加载。加载COM加载项时出现运行错误"。
> 这是因为 RegAsm 不会自动在 CLSID 下添加 `AppID` 属性，而 OneNote 的 COM 激活机制依赖此属性。

#### 步骤 1：使用 RegAsm 注册 COM 类型

以 **管理员身份** 打开命令提示符，执行以下命令：

```cmd
:: 使用完整路径调用 RegAsm（它不在系统 PATH 中）
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\RegAsm.exe /codebase OneMarkDotNet.dll
```

> **重要**：因为项目目标平台为 x64，必须使用 `Framework64` 目录下的 RegAsm.exe，不能使用 `Framework`（32位）目录下的版本。
>
> 如果你的系统上找不到 RegAsm.exe，请确认已安装 .NET Framework 4.8。RegAsm.exe 随 .NET Framework 自动安装，无需单独下载。常见路径：
> - 64位：`C:\Windows\Microsoft.NET\Framework64\v4.0.30319\RegAsm.exe`
> - 32位：`C:\Windows\Microsoft.NET\Framework\v4.0.30319\RegAsm.exe`

也可以先 cd 到输出目录再注册：

```cmd
cd src\OneMarkDotNet\bin\Debug
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\RegAsm.exe /codebase OneMarkDotNet.dll
```

注册成功后会看到类似输出：

```
Microsoft .NET Framework 程序集注册实用工具版本 4.8.xxxx.x
版权所有(C) Microsoft Corporation。保留所有权利。
RegAsm : warning RA0000 : 使用 /codebase 注册未签名的程序集...
成功注册了类型
```

> **注意**：`/codebase` 参数会在注册表中记录 DLL 的完整路径，使 OneNote 能找到该 DLL。如果 DLL 路径发生变化（如切换 Debug/Release 目录），需要重新注册。RA0000 警告可以忽略，仅提示程序集未签名。

#### 步骤 2：导入完整注册表文件

**这一步是关键，不能跳过！** RegAsm 不会生成 OneNote 所需的 `AppID` 注册项。

项目提供了 `register_full.reg` 文件，双击导入即可：

```
src\OneMarkDotNet\register_full.reg    ← 完整注册（推荐）
src\OneMarkDotNet\unregister.reg       ← 注销
```

或使用命令行导入：

```cmd
reg import "E:\Project\NoteMark\src\OneMarkDotNet\register_full.reg"
```

`register_full.reg` 包含以下关键注册项（RegAsm 不会自动生成的部分）：

| 注册表项 | 作用 | 说明 |
|---------|------|------|
| `HKCR\CLSID\{GUID}\AppID` | COM 激活机制 | **最关键**，OneNote 依赖此属性激活 .NET 组件 |
| `HKCR\AppID\{GUID}\DllSurrogate` | COM 代理进程 | 允许 COM 在代理进程中激活 |
| `HKCR\OneMarkDotNet.AddIn\CurVer` | ProgID 版本引用 | COM 标准注册结构 |
| `HKCR\OneMarkDotNet.AddIn.1` | 版本化 ProgID | COM 标准注册结构 |
| `HKCU\...\OneNote\AddIns\OneMarkDotNet.AddIn` | OneNote 加载项条目 | LoadBehavior=3 启动时加载 |
| `HKLM\...\OneNote\AddIns\OneMarkDotNet.AddIn` | 系统级加载项条目 | 所有用户可见 |
| `HKCU\...\Resiliency\DoNotDisableAddinList` | 防止 OneNote 禁用 | 避免加载失败后被自动禁用 |

#### 步骤 3：重启 OneNote

关闭所有 OneNote 进程后重新打开：

```cmd
taskkill /f /im ONENOTE.EXE
```

然后从开始菜单或桌面启动 OneNote。

### 2.2 注销 Add-In

```cmd
:: 1. 注销 COM 类型
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\RegAsm.exe /u OneMarkDotNet.dll

:: 2. 导入注销注册表文件
reg import "E:\Project\NoteMark\src\OneMarkDotNet\unregister.reg"
```

### 2.3 验证注册表

注册完成后，检查以下注册表项：

**OneNote Add-In 条目**：

```
HKEY_CURRENT_USER\Software\Microsoft\Office\OneNote\AddIns\OneMarkDotNet.AddIn
```

| 名称 | 类型 | 值 | 说明 |
|------|------|----|------|
| FriendlyName | REG_SZ | OneMarkDotNet | 显示名称 |
| Description | REG_SZ | OneNote Markdown Extension | 描述 |
| LoadBehavior | REG_DWORD | 3 | 启动时加载 |

**CLSID 注册（关键 — 确认 AppID 存在）**：

```
HKEY_CLASSES_ROOT\CLSID\{B2C3D4E5-F6A7-8901-BCDE-F12345678901}
```

| 名称 | 值 | 说明 |
|------|----|------|
| (Default) | OneMarkDotNet.OneMarkAddIn | 类名 |
| **AppID** | **{B2C3D4E5-F6A7-8901-BCDE-F12345678901}** | **必须存在！** |

**AppID 注册**：

```
HKEY_CLASSES_ROOT\AppID\{B2C3D4E5-F6A7-8901-BCDE-F12345678901}
```

| 名称 | 值 | 说明 |
|------|----|------|
| (Default) | OneMarkDotNet.OneMarkAddIn | 类名 |
| DllSurrogate | (空字符串) | COM 代理 |

> **排查提示**：如果 OneNote 报"运行错误"，首先检查 CLSID 下是否有 `AppID` 属性。对比系统上正常运行的 OneMore Add-In（`{88AB88AB-CDFB-4C68-9C3A-F10B75A5BC61}`）的注册结构，确认我们的注册项与其一致。

**LoadBehavior 值说明**：

| 值 | 含义 |
|----|------|
| 0 | 禁用，不加载 |
| 1 | 仅在用户手动启用时加载 |
| 2 | 启动时加载，但失败时不重试 |
| 3 | 启动时加载（推荐） |

## 3. F5 直接调试（推荐）

项目已配置好调试启动设置，在 Visual Studio 中直接按 **F5** 即可启动 OneNote 并附加调试器。

### 3.1 前置条件

1. 以 **管理员身份** 运行 Visual Studio（构建后自动注册 COM 需要）
2. 确保已执行过一次完整的注册流程（步骤 1 + 步骤 2，见第 2 节）
3. 关闭所有 OneNote 窗口

### 3.2 调试步骤

1. 用 Visual Studio 打开 `OneMarkDotNet.sln`
2. 确认配置为 **Debug | x64**
3. 在代码中设置断点（如 `OneMarkAddIn.cs` 的 `OnConnection` 方法）
4. 按 **F5** 启动调试

Visual Studio 会自动：
- 构建 Debug 版本
- 执行 RegAsm 注册 COM 类型
- 创建 OneNote Add-In 注册表条目
- 启动 OneNote（`ONENOTE.EXE`）并附加调试器

> **注意**：每次构建后 RegAsm 可能覆盖 AppID 注册项。如果 F5 启动后 OneNote 报"运行错误"，需要重新导入 `register_full.reg`，然后重启 OneNote。

### 3.3 .csproj.user 配置说明

项目已包含 `OneMarkDotNet.csproj.user` 文件，配置了以下调试参数：

```xml
<Project ToolsVersion="15.0" xmlns="http://schemas.microsoft.com/developer/msbuild/2003">
  <PropertyGroup Condition="'$(Configuration)|$(Platform)' == 'Debug|x64'">
    <StartAction>Program</StartAction>
    <StartProgram>C:\Program Files\Microsoft Office\root\Office16\ONENOTE.EXE</StartProgram>
    <EnableUnmanagedDebugging>true</EnableUnmanagedDebugging>
  </PropertyGroup>
</Project>
```

- `StartAction=Program`：启动外部程序而非尝试运行类库
- `StartProgram`：OneNote 可执行文件路径
- `EnableUnmanagedDebugging=true`：启用非托管代码调试，可以调试 COM 互操作调用

> **注意**：如果你的 OneNote 安装路径不同，需要修改 `StartProgram` 的值。Microsoft 365 版本通常在 `C:\Program Files\Microsoft Office\root\Office16\`，独立版 OneNote 2016 可能在 `C:\Program Files (x86)\Microsoft Office\Office16\`。

### 3.4 构建后自动注册

csproj 中已配置 PostBuild 目标，每次构建后自动执行：
1. `RegAsm.exe /codebase OneMarkDotNet.dll` — 注册 COM 类型
2. `reg add HKCU\...\OneMarkDotNet.AddIn` — 创建 OneNote Add-In 注册表条目

如果不需要自动注册（如只做代码修改不调试），可以禁用：

```cmd
msbuild /p:PostBuildRegAsm=false
```

## 4. 手动附加调试

如果 F5 调试不适用（如 OneNote 已经在运行），可以手动附加调试器。

### 步骤 1：构建 Debug 版本

确保项目以 Debug 配置构建，DLL 包含调试信息（.pdb 文件）。

### 步骤 2：注册 COM Add-In

按第 2 节的完整流程执行（RegAsm + register_full.reg）。

### 步骤 3：打开 Visual Studio，附加到进程

在 Visual Studio 中选择 **调试** → **附加到进程**。

### 步骤 4：找到 ONENOTE.EXE

在进程列表中找到 `ONENOTE.EXE`。如果列表较长，可以在筛选框中输入 `onenote` 快速定位。

> **提示**：确保勾选了"显示所有用户的进程"，否则可能看不到 OneNote 进程。

### 步骤 5：确保附加为托管代码类型

点击"附加到进程"对话框中的"选择"按钮，选择 **托管(v4.6)** 代码类型。不要选择"自动"，因为 OneNote 是混合进程（原生 + 托管），自动检测可能附加为原生代码类型，导致断点无法命中。

### 步骤 6：设置断点

在代码中设置断点，例如：
- `OneMarkAddIn.cs` 的 `OnConnection` 方法 — 确认 Add-In 是否被加载
- `MarkdownRenderHandler.cs` 的 `HandleRenderSelection` 方法 — 调试渲染流程
- `MarkdownParser.cs` 的 `Parse` 方法 — 调试解析逻辑

### 步骤 7：触发操作

在 OneNote 中执行相应操作触发断点，例如：
- 重启 OneNote 触发 `OnConnection`
- 点击 Ribbon 按钮触发渲染
- 使用快捷键触发实时渲染

## 5. 查看日志

### 5.1 日志位置

| 项目 | 值 |
|------|----|
| 日志目录 | `%APPDATA%\OneMarkDotNet\logs\` |
| 日志文件名 | `onemark_yyyyMMdd.log` |
| 日志格式 | `[时间] [级别] 消息` |
| 自动清理 | 30 天前的日志自动删除 |

### 5.2 日志格式示例

```
[2026-06-11 14:30:25] [INFO] Add-In 已加载: OnConnection
[2026-06-11 14:30:26] [INFO] Ribbon UI 已初始化
[2026-06-11 14:31:05] [DEBUG] 开始渲染选区
[2026-06-11 14:31:05] [DEBUG] Markdown 解析完成: 3 个块级元素
[2026-06-11 14:31:06] [ERROR] COM 调用失败: 0x80004005 - 未指定的错误
```

### 5.3 常用查看方式

打开日志目录：

```cmd
explorer %APPDATA%\OneMarkDotNet\logs
```

实时查看日志（类似 Linux 的 `tail -f`）：

```cmd
powershell Get-Content "$env:APPDATA\OneMarkDotNet\logs\onemark_$(Get-Date -Format yyyyMMdd).log" -Wait -Tail 50
```

查看最近的错误日志：

```cmd
powershell Select-String -Path "$env:APPDATA\OneMarkDotNet\logs\onemark_$(Get-Date -Format yyyyMMdd).log" -Pattern "\[ERROR\]"
```

## 6. 常见调试场景

### 场景 1：Add-In 未加载 / "运行错误"

**症状**：OneNote 启动后看不到 OneMarkDotNet 的 Ribbon 按钮，或报"未加载。加载COM加载项时出现运行错误"。

**排查步骤**：

1. **检查 CLSID 下是否有 AppID 属性（最常见原因）**
   - 打开 `regedit`，导航到 `HKCR\CLSID\{B2C3D4E5-F6A7-8901-BCDE-F12345678901}`
   - 检查是否有 `AppID` 值，指向 `{B2C3D4E5-F6A7-8901-BCDE-F12345678901}`
   - 如果没有，重新导入 `register_full.reg`
   - **这是 OneNote COM Add-In 加载失败最常见的原因**：RegAsm 不会自动生成 AppID

2. **检查注册表项是否存在**
   - 打开 `regedit`，导航到 `HKEY_CURRENT_USER\Software\Microsoft\Office\OneNote\AddIns\OneMarkDotNet.AddIn`
   - 如果不存在，说明注册未成功，重新执行完整注册流程

3. **检查 LoadBehavior 是否为 3**
   - 如果为 0 或 2，OneNote 不会在启动时加载 Add-In
   - 手动修改为 3 或重新导入 `register_full.reg`

4. **检查 .NET Framework 4.8 是否安装**
   - 打开 `C:\Windows\Microsoft.NET\Framework64\v4.0.30319` 目录确认存在
   - 或在注册表 `HKLM\SOFTWARE\Microsoft\NET Framework Setup\NDP\v4\Full` 中检查 `Release` 值 ≥ 528040

5. **检查 OneNote 信任中心是否允许加载 Add-In**
   - OneNote 中：**文件** → **选项** → **信任中心** → **信任中心设置** → **加载项**
   - **取消勾选**"禁用所有应用程序加载项"
   - 如果启用了此选项，所有 COM Add-In 都不会被加载

6. **检查 OneNote 加载项列表**
   - OneNote 中：**文件** → **选项** → **加载项**
   - 在"活动的应用程序加载项"列表中查找 OneMarkDotNet
   - 如果在"非活动的应用程序加载项"中，说明加载失败或被禁用
   - 如果在"已禁用的应用程序加载项"中，点击"转到"启用它

7. **对比正常工作的 Add-In 注册结构**
   - 系统上如果有 OneMore（`River.OneMoreAddIn`）或 Neux.OneMark，对比它们的 CLSID 注册
   - 正常工作的 Add-In 一定有 `AppID` 属性和 `HKCR\AppID\{GUID}\DllSurrogate`

---

### 场景 2：Ribbon 不显示

**症状**：Add-In 已加载（日志中有 OnConnection 记录），但 Ribbon 上看不到按钮。

**排查步骤**：

1. **检查 Ribbon.xml 是否正确嵌入为资源**
   - 在 Visual Studio 中打开 `Ribbon.xml` 属性
   - 确认 **生成操作** 为 `嵌入的资源`
   - 如果不是，修改后重新构建

2. **检查 GetCustomUI 方法是否返回正确的 XML**
   - 在 `OneMarkRibbon.cs` 的 `GetCustomUI` 方法中设置断点
   - 确认返回的 XML 字符串不为空且格式正确
   - 确认 `ribbonID` 参数为 `Microsoft.OneNote.4`（OneNote 的 Ribbon ID）

3. **检查 ProgId 是否与注册表一致**
   - `OneMarkAddIn.cs` 中的 `[ProgId("OneMarkDotNet.AddIn")]` 特性值
   - 必须与注册表项名 `HKEY_CURRENT_USER\...\AddIns\OneMarkDotNet.AddIn` 一致

4. **在 OnConnection 中设置断点确认是否被调用**
   - 如果 `OnConnection` 被调用但 `GetCustomUI` 未被调用，可能是 Ribbon ID 不匹配

---

### 场景 3：快捷键不响应

**症状**：Ribbon 按钮可以正常使用，但快捷键（如 Ctrl+M）无反应。

**排查步骤**：

1. **检查 KeyboardHook 是否成功安装**
   - 在 `KeyboardHook.cs` 的 `SetWindowsHookEx` 调用后检查返回值
   - 如果返回 `IntPtr.Zero`，说明钩子安装失败
   - 使用 `Marshal.GetLastWin32Error()` 获取错误码

2. **检查 OneNote 窗口是否为前台窗口**
   - 全局键盘钩子只在特定条件下触发
   - 确认 OneNote 窗口处于活动状态

3. **检查 AddInSettings 中 IsRealtimeRenderEnabled 是否为 true**
   - 如果实时渲染被禁用，快捷键可能不会触发渲染流程
   - 在 `AddInSettings.cs` 中检查默认值和当前值

4. **在 KeyboardHook 的回调方法中设置断点**
   - 确认回调是否被触发
   - 检查按键组合是否匹配预期

---

### 场景 4：渲染结果不正确

**症状**：Markdown 渲染后输出与预期不符，如样式丢失、元素缺失、格式错乱。

**排查步骤**：

1. **在 MarkdownRenderHandler.HandleRenderSelection 中设置断点**
   - 这是渲染流程的入口，可以追踪整个渲染管线
   - 检查输入的 Markdown 文本是否正确获取

2. **检查 MarkdownParser.Parse 的输出**
   - 检查解析后的 `MarkdownDocument` 对象
   - 确认 `Blocks` 和 `Inlines` 集合中的元素类型和内容是否正确
   - 常见问题：Markdig 扩展未注册导致特定语法未被解析

3. **检查 OneNoteXmlConverter.ConvertToOneNoteXml 的输出**
   - 检查生成的 OneNote XML 是否符合预期
   - 特别注意样式属性的映射是否正确
   - 可以将生成的 XML 输出到日志进行离线分析

4. **使用 OneNoteXmlBuilder 单独测试 XML 生成**
   - 将 OneNoteXmlBuilder 的输出与 OneNote 期望的 XML 格式对比
   - OneNote 的 XML Schema 文档：在 OneNote 中导出页面为 `.one` 文件后解压查看

---

### 场景 5：COM 异常

**症状**：运行时抛出 `COMException`，错误码如 `0x8002802B`、`0x80004005` 等。

**排查步骤**：

1. **所有 COM 调用都应包裹在 try-catch 中**
   - 检查异常的 `ErrorCode` 属性获取 HRESULT
   - 检查 `Message` 属性获取错误描述

2. **检查 OneNoteApiWrapper 中的 dynamic 调用**
   - `dynamic` 类型的 COM 调用无法在编译时检查
   - 使用 `dynamic` 前先检查对象是否为 `null`
   - 考虑将频繁调用的 `dynamic` 替换为强类型接口

3. **常见 COM 错误码**

   | 错误码 | 含义 | 常见原因 |
   |--------|------|----------|
   | 0x8002802B | 类型不匹配 | 传递了错误类型的参数 |
   | 0x80004005 | 未指定错误 | OneNote 内部错误，检查日志获取更多信息 |
   | 0x80070005 | 访问被拒绝 | 权限不足，尝试以管理员身份运行 |
   | 0x80070057 | 参数不正确 | 传递了无效的参数值 |
   | 0x8001010A | 消息筛选器指示应用程序正在忙碌 | OneNote 正忙，需要重试 |

4. **使用 Marshal.GetLastWin32Error() 获取详细错误**
   - 在 COM 调用后立即调用
   - 注意：只有在 COM 方法返回错误时才有意义

## 7. 调试技巧

### 使用 Debugger.Launch()

在 `OneMarkAddIn.cs` 的 `OnConnection` 方法中添加以下代码：

```csharp
System.Diagnostics.Debugger.Launch();
```

OneNote 启动并加载 Add-In 时，会弹出调试器选择对话框，选择 Visual Studio 实例即可自动附加。这在无法手动附加调试器时特别有用。

> **注意**：调试完成后务必删除此行，否则每次启动 OneNote 都会弹出对话框。

### 使用 Debug.WriteLine

在关键路径添加 `Debug.WriteLine` 输出到 Visual Studio 的输出窗口：

```csharp
System.Diagnostics.Debug.WriteLine($"[OneMark] 解析完成: {doc.Blocks.Count} 个块级元素");
```

此方法仅在 Debug 构建中生效，不会影响 Release 构建的性能。

### 使用 AppLogger

所有日志通过 `AppLogger.Instance` 记录，可实时查看：

```csharp
AppLogger.Instance.Debug("调试信息");
AppLogger.Instance.Info("常规信息");
AppLogger.Instance.Warning("警告信息");
AppLogger.Instance.Error("错误信息", exception);
```

AppLogger 同时输出到文件和 Debug 窗口，是调试时的首选日志工具。

### 修改后重新注册

每次修改代码后需要执行以下完整流程：

```cmd
:: 1. 关闭 OneNote（DLL 被锁定，必须先关闭）
taskkill /f /im ONENOTE.EXE

:: 2. 重新构建（在 Visual Studio 中或命令行）
msbuild OneMarkDotNet.sln /t:Build /p:Configuration=Debug /p:Platform=x64

:: 3. 重新注册 COM 类型（构建后自动执行，也可手动）
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\RegAsm.exe /codebase OneMarkDotNet.dll

:: 4. 导入完整注册表（补充 AppID 等关键项）
reg import "E:\Project\NoteMark\src\OneMarkDotNet\register_full.reg"

:: 5. 重新打开 OneNote
```

> **提示**：步骤 3 和 4 也可以合并为双击 `register_full.reg`，因为它包含所有需要的注册项。但 RegAsm 仍然需要先执行，因为它负责生成 CLSID 下的 InprocServer32、Assembly 等基础信息。

### 清理 OneNote 缓存

有时 OneNote 会缓存 Add-In 的状态，导致修改不生效。需要关闭所有 OneNote 进程再重新打开：

```cmd
taskkill /f /im ONENOTE.EXE
```

如果 OneNote 进程无法正常关闭，可以使用：

```cmd
taskkill /f /im ONENOTE.EXE /t
```

`/t` 参数会同时关闭 OneNote 的所有子进程。

### 检查 .NET 加载版本

如果遇到 `FileNotFoundException` 或 `FileLoadException`，可能是程序集绑定失败。使用 **Fusion Log Viewer** (fuslogvw) 检查：

1. 以管理员身份打开 **Developer Command Prompt**
2. 运行 `fuslogvw`
3. 点击"设置" → 选择"记录绑定失败到磁盘"
4. 重现问题
5. 点击"刷新"查看绑定失败日志

常见原因：
- DLL 版本不匹配
- .NET Framework 版本不正确
- NuGet 包未正确还原

## 8. 性能分析

### 8.1 使用 Visual Studio 性能探查器

1. 打开 Visual Studio → **调试** → **性能探查器**
2. 选择 **CPU 使用率** 或 **检测**
3. 附加到 `ONENOTE.EXE` 进程
4. 在 OneNote 中执行渲染操作
5. 停止收集，分析结果

### 8.2 关注的关键性能区域

| 区域 | 关注点 | 优化方向 |
|------|--------|----------|
| MarkdownParser.Parse | 解析耗时 | 缓存 Pipeline 实例，避免重复创建 |
| OneNoteXmlConverter.ConvertToOneNoteXml | 转换耗时 | 减少 XML 字符串拼接，使用 StringBuilder |
| LatexRenderService | 首次调用延迟 | WebView2 初始化是性能瓶颈，考虑预初始化 |
| DiagramRenderService | 首次调用延迟 | 同上，WebView2 冷启动约需 1-3 秒 |
| OneNoteApiWrapper | COM 调用延迟 | 减少 COM 跨进程调用次数，批量操作 |

### 8.3 WebView2 性能注意事项

WebView2 的初始化是最大的性能瓶颈：

- **首次初始化**：约 1-3 秒，包括创建 WebView2 环境、启动浏览器进程
- **后续调用**：约 50-200ms，取决于渲染内容的复杂度
- **建议**：在 `OnConnection` 时预初始化 WebView2 环境，避免用户首次操作时的延迟

## 9. 单元测试

以下组件不依赖 OneNote COM，可以独立编写单元测试：

### 9.1 可独立测试的组件

| 组件 | 测试内容 | 依赖 |
|------|----------|------|
| MarkdownParser | Markdown 文本 → MarkdownDocument | Markdig |
| MarkdownDocument.ConvertBlock / ConvertInline | 元素模型转换 | 无 |
| OneNoteXmlBuilder | OneNote XML 生成 | 无 |
| FrontMatterParser | Front Matter 解析 | 无 |
| CodeHighlightService | 代码高亮规则 | 无 |
| CssVariableParser | CSS 变量解析 | 无 |

### 9.2 示例测试代码

```csharp
using Microsoft.VisualStudio.TestTools.UnitTesting;
using OneMarkDotNet;

namespace OneMarkDotNet.Tests
{
    [TestClass]
    public class MarkdownParserTests
    {
        private MarkdownParser _parser;

        [TestInitialize]
        public void Setup()
        {
            _parser = new MarkdownParser();
        }

        [TestMethod]
        public void Parse_Heading_ReturnsHeadingElement()
        {
            // Arrange
            var markdown = "# Hello";

            // Act
            var doc = _parser.Parse(markdown);

            // Assert
            Assert.AreEqual(1, doc.Blocks.Count);
            Assert.IsInstanceOfType(doc.Blocks[0], typeof(HeadingElement));
        }

        [TestMethod]
        public void Parse_BoldAndNormalText_ReturnsCorrectElements()
        {
            // Arrange
            var markdown = "# Hello\n\n**bold** text";

            // Act
            var doc = _parser.Parse(markdown);

            // Assert
            Assert.AreEqual(2, doc.Blocks.Count);
            Assert.IsInstanceOfType(doc.Blocks[0], typeof(HeadingElement));
            Assert.IsInstanceOfType(doc.Blocks[1], typeof(ParagraphElement));
        }

        [TestMethod]
        public void Parse_CodeBlock_ReturnsCodeBlockElement()
        {
            // Arrange
            var markdown = "```csharp\nConsole.WriteLine();\n```";

            // Act
            var doc = _parser.Parse(markdown);

            // Assert
            Assert.AreEqual(1, doc.Blocks.Count);
            Assert.IsInstanceOfType(doc.Blocks[0], typeof(CodeBlockElement));
        }
    }
}
```

### 9.3 测试注意事项

- **不要在单元测试中调用 OneNote COM API**，这些调用需要 OneNote 运行
- 对于依赖 WebView2 的组件（LatexRenderService、DiagramRenderService），使用集成测试而非单元测试
- 测试 Markdown 解析时，覆盖边界情况：空输入、纯文本、嵌套语法、非法 Markdown 等

## 10. 故障排除清单

当 Add-In 出现问题时，按以下清单逐项排查：

- [ ] **CLSID 下有 AppID 属性（最常见遗漏项）**
  - 检查 `HKCR\CLSID\{B2C3D4E5-F6A7-8901-BCDE-F12345678901}\AppID` 是否存在
  - 值应为 `{B2C3D4E5-F6A7-8901-BCDE-F12345678901}`
  - 如果缺失，导入 `register_full.reg`

- [ ] **AppID 注册项存在**
  - 检查 `HKCR\AppID\{B2C3D4E5-F6A7-8901-BCDE-F12345678901}\DllSurrogate` 是否存在

- [ ] **.NET Framework 4.8 已安装**
  - 检查注册表 `HKLM\SOFTWARE\Microsoft\NET Framework Setup\NDP\v4\Full\Release` ≥ 528040

- [ ] **WebView2 Runtime 已安装**
  - 检查 `C:\Program Files (x86)\Microsoft\EdgeUpdate` 或 Edge 浏览器版本

- [ ] **RegAsm 注册成功（无错误输出）**
  - 以管理员身份运行，确认输出"类型已成功注册"

- [ ] **注册表项存在且 LoadBehavior = 3**
  - 路径：`HKCU\Software\Microsoft\Office\OneNote\AddIns\OneMarkDotNet.AddIn`

- [ ] **DLL 路径正确（codebase 指向正确位置）**
  - 检查 `HKCR\CLSID\{...}\InprocServer32\CodeBase` 值
  - 路径必须与实际 DLL 位置一致

- [ ] **OneNote 信任中心允许加载项**
  - 文件 → 选项 → 信任中心 → 信任中心设置 → 加载项
  - 取消勾选"禁用所有应用程序加载项"

- [ ] **日志文件中有 OnConnection 记录**
  - 检查 `%APPDATA%\OneMarkDotNet\logs\` 目录
  - 如果没有日志文件，说明 Add-In 根本未被加载

- [ ] **无其他 OneNote Add-In 冲突**
  - 在 OneNote 的加载项列表中检查是否有其他 Add-In
  - 尝试禁用其他 Add-In 后重新测试
  - 某些 Add-In 可能共享相同的依赖导致版本冲突

---

> **最后建议**：调试 COM Add-In 是一项需要耐心的工作。遇到问题时，先查看日志，再按故障排除清单逐项排查，大多数问题都能定位到原因。如果遇到无法解决的问题，请在项目仓库提交 Issue，附上日志文件和复现步骤。
