# OneMarkDotNet 架构设计

## 概述

OneMarkDotNet 采用分层模块化架构。虽然项目编译为单一程序集（OneMarkDotNet.dll），但代码按职责清晰地分为 6 个逻辑层，各层之间通过接口和事件进行解耦：

1. **AddIn 层** — COM 入口与用户交互
2. **Handler 层** — 业务流程编排
3. **Converter 层** — 格式双向转换
4. **Rendering 层** — 专项内容渲染
5. **Engine 层** — Markdown 解析与文档模型
6. **Infrastructure 层** — 主题、日志、设置等基础设施

## 各模块详解

### AddIn 层

| 类 | 职责 |
|----|------|
| `OneMarkAddIn` | COM Add-In 入口，实现 `IDTExtensibility2` 接口，负责插件的生命周期管理（连接、断开）和初始化 |
| `KeyboardHook` | 全局键盘钩子，使用 `WH_KEYBOARD_LL` 捕获按键事件，触发渲染/导出/源码模式等操作 |
| `OneMarkRibbon` | Ribbon 自定义按钮，提供图形化操作入口（导入、导出、主题切换等） |

### Handler 层

| 类 | 职责 |
|----|------|
| `MarkdownRenderHandler` | 渲染流程控制器，协调解析、转换、更新页面的完整流程 |
| `ExportHandler` | 导入导出流程控制器，处理 Markdown 导出和导入操作 |

### Converter 层

| 类 | 职责 |
|----|------|
| `OneNoteXmlConverter` | OneNote XML 双向转换核心，负责 XML 的解析和生成 |
| `HtmlToOneNoteConverter` | 将 Markdown 解析结果转换为 OneNote XML 结构 |
| `OneNoteToMarkdownConverter` | 将 OneNote 页面 XML 反向转换为 Markdown 文本 |

### Rendering 层

| 类 | 职责 |
|----|------|
| `CodeHighlightService` | 代码语法高亮服务，支持 17 种编程语言，将代码块转换为带样式的 OneNote 内容 |
| `LatexRenderService` | LaTeX 公式渲染服务，通过 WebView2 离屏渲染数学公式为图片 |
| `DiagramRenderService` | Mermaid 图表渲染服务，通过 WebView2 离屏渲染图表为图片 |

### Engine 层

| 类 | 职责 |
|----|------|
| `MarkdownParser` | Markdown 解析器，基于 Markdig 管道（Pipeline）构建语法树 |
| `MarkdownDocument` | 文档模型，管理元素树，提供元素的增删查改 |
| `MarkdownRenderer` | 渲染器，将文档模型渲染为 HTML 或 OneNote XML |
| `MarkdownExtension` | Markdig 扩展，注册数学公式（`$...$` / `$$...$$`）和 Mermaid 图表语法 |
| `Elements/*` | 20 个元素模型类，表示 Markdown 的各种语法元素 |

### Infrastructure 层

| 类 | 职责 |
|----|------|
| `OneNoteApiWrapper` | OneNote COM API 封装，提供页面读写、层级操作等高层接口 |
| `OneNotePageUpdater` | 页面内容更新器，负责将渲染结果写回 OneNote 页面 |
| `OneNoteXmlBuilder` | OneNote XML 构建工具，提供流式 API 生成 XML |
| `ThemeManager` | 主题管理器，加载和切换 CSS 主题 |
| `AppLogger` | 异步日志，基于 `ConcurrentQueue` + `Timer` 实现非阻塞日志写入 |
| `AddInSettings` | 用户设置管理，持久化插件配置 |
| `WebView2Helper` | WebView2 环境初始化辅助，管理 WebView2 生命周期 |

## 元素模型

OneMarkDotNet 使用面向对象的元素模型表示 Markdown 文档结构，分为块级元素和行内元素两大类。

### 继承层次

```
MarkdownElement (抽象基类)
├── BlockElement (块级元素基类)
│   ├── HeadingElement          # 标题 (H1-H6)
│   ├── ParagraphElement        # 段落
│   ├── CodeBlockElement        # 代码块 (```)
│   ├── QuoteBlockElement       # 引用块 (>)
│   ├── ListElement             # 有序/无序列表
│   ├── TaskListElement         # 任务列表 (- [ ] / - [x])
│   ├── TableElement            # 表格
│   ├── HorizontalRuleElement   # 分隔线 (---)
│   ├── MathBlockElement        # 块级数学公式 ($$...$$)
│   └── DiagramBlockElement     # Mermaid 图表
└── InlineElement (行内元素基类)
    ├── TextElement             # 纯文本
    ├── BoldElement             # 加粗 (**text**)
    ├── ItalicElement           # 斜体 (*text*)
    ├── StrikethroughElement    # 删除线 (~~text~~)
    ├── CodeInlineElement       # 行内代码 (`code`)
    ├── LinkElement             # 链接 [text](url)
    ├── ImageElement            # 图片 ![alt](url)
    └── MathInlineElement       # 行内数学公式 ($...$)
```

### 设计特点

- **双基类分离**：`BlockElement` 和 `InlineElement` 分离，确保类型安全
- **组合模式**：`ParagraphElement`、`ListElement` 等块级元素可包含多个 `InlineElement`
- **统一接口**：所有元素继承自 `MarkdownElement`，提供统一的 `Accept` 方法供访问者模式使用

## 关键设计决策

### 1. Markdig 作为解析基础

选择 Markdig 而非自行实现 Markdown 解析器，原因：

- Markdig 是 .NET 生态最成熟的 Markdown 解析库，遵循 CommonMark 规范
- 支持扩展管道（Pipeline），便于添加自定义语法（数学公式、Mermaid）
- 生成结构化语法树，可直接映射到元素模型

通过 `MarkdownExtension` 注册自定义扩展，在 Markdig 管道中无缝集成数学公式和图表语法。

### 2. WebView2 离屏渲染 LaTeX / 图表

LaTeX 公式和 Mermaid 图表无法直接用 OneNote 原生元素表示，因此采用 WebView2 离屏渲染策略：

- 创建不可见的 WebView2 实例
- 加载包含 MathJax / Mermaid.js 的 HTML 页面
- 将渲染结果捕获为图片
- 将图片插入 OneNote 页面

`WebView2Helper` 管理 WebView2 环境的初始化和生命周期，`LatexRenderService` 和 `DiagramRenderService` 分别封装各自的渲染逻辑。

### 3. 手动定义 COM 接口（无 VSTO 依赖）

不使用 VSTO（Visual Studio Tools for Office），而是手动定义 COM 接口：

- 直接实现 `IDTExtensibility2` 接口
- 手动定义 OneNote COM Interop 接口
- 通过 `regasm /codebase` 注册 COM 组件

优势：无需安装 VSTO Runtime，减少部署依赖；对 COM 接口有完全控制权。

### 4. 异步日志（ConcurrentQueue + Timer）

`AppLogger` 采用异步日志设计，避免 I/O 操作阻塞渲染流程：

- 日志消息入队到 `ConcurrentQueue`（线程安全，无锁快速入队）
- `Timer` 定期批量出队并写入文件
- 渲染线程不受磁盘 I/O 影响

### 5. CSS 变量驱动的主题系统

主题系统基于 CSS 变量实现，而非硬编码样式：

- `__global.css` 定义默认 CSS 变量（颜色、字号、间距等）
- 各主题文件覆盖需要变更的变量
- `CssVariableParser` 解析 CSS 变量
- `CssStyleMapper` 将 CSS 属性映射为 OneNote 原生样式
- `ConverterOneNoteStyle` 执行最终的样式转换

这种设计使得添加新主题只需编写 CSS 文件，无需修改 C# 代码。

### 6. 全局键盘钩子（WH_KEYBOARD_LL）

使用 Windows 底层键盘钩子而非 OneNote 内置快捷键机制：

- `KeyboardHook` 通过 `SetWindowsHookEx(WH_KEYBOARD_LL, ...)` 安装全局钩子
- 可捕获 OneNote 窗口内的所有按键事件
- 支持组合键（Ctrl+,、Ctrl+Enter）和功能键（F5、F8）
- 钩子在 `OneMarkAddIn.OnConnection` 中安装，在 `OnDisconnection` 中卸载

## OneNote XML 命名空间

OneMarkDotNet 操作的 OneNote XML 使用以下命名空间：

```
http://schemas.microsoft.com/office/onenote/2013/onenote
```

该命名空间对应 OneNote 2013 及更高版本的 XML Schema，所有 XML 操作（读取页面、更新内容、构建元素）均基于此命名空间。`OneNoteXmlConverter`、`OneNoteXmlBuilder`、`HtmlToOneNoteConverter` 等类在生成 XML 时必须使用正确的命名空间前缀，以确保 OneNote 能正确解析。
